# Implementation Plan: Drag-and-Drop Session Builder

**Branch**: `026-drag-drop-builder` | **Date**: 2026-02-18 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/026-drag-drop-builder/spec.md`

## Summary

Replace the up/down arrow reordering interaction with drag-and-drop in the setlist builder and routine edit page, and make library item rows fully tappable. This is a **shell-only change** — no core logic, API, or data model modifications are needed. The existing `ReorderSetlist` core event and the routine edit page's local signal-based reordering are reused as-is. The implementation uses the Pointer Events API for cross-device support (desktop mouse + mobile touch), with drag initiation restricted to a dedicated drag handle to preserve scroll on touch devices.

## Key Design Decisions

### 1. Pointer Events over HTML5 Drag and Drop

The HTML5 DnD API does not work on iOS Safari touch interactions (dragstart never fires) and has inconsistent Android support. The Pointer Events API (`pointerdown`/`pointermove`/`pointerup`) provides a unified model across mouse, touch, and stylus with excellent browser support (iOS Safari 13+).

### 2. Reusable Hook Pattern

Drag-and-drop logic is shared between the session builder (which dispatches `ReorderSetlist` core events) and the routine edit page (which updates a local `RwSignal<Vec<RoutineEntryView>>`). A `use_drag_reorder` hook encapsulates the drag state management and position-calculation logic, accepting a callback for the actual reorder operation. This avoids duplicating ~100 lines of pointer event handling.

### 3. No Visual Clone / Ghost Element

During drag, the source item receives a visual highlight (opacity, border) and a drop indicator line appears between entries at the computed target position. The item is NOT visually moved with the pointer (no `translateY`). This significantly simplifies implementation, avoids layout-shift bugs, and matches the interaction pattern used by modern reorder UIs (Notion, Linear).

### 4. Drop Indicator via Conditional Rendering

The drop indicator is rendered as a conditional `<div>` between list items based on the `hover_index` signal. This is simpler than CSS pseudo-element toggling in Leptos and allows the indicator to be a first-class component (`DropIndicator`) with consistent styling.

### 5. Dependency on 025-reusable-routines

FR-010 (drag-and-drop on routine edit page) requires the routine edit view from feature 025. The 026 branch must rebase onto main after 025 is merged. If 025 is not merged when implementation begins, FR-010 tasks can be deferred without blocking P1/P2 stories.

### 6. web-sys Feature Additions

The following web-sys features are added to intrada-web's Cargo.toml:
- `PointerEvent` — for pointer event handlers
- `Element` — for `get_bounding_client_rect()`, `closest()`
- `DomRect` — return type of bounding rect
- `HtmlElement` — for `set_pointer_capture()`, `release_pointer_capture()`

Existing features (`Window`, `Storage`, `ClipboardEvent`, `DataTransfer`) are preserved.
