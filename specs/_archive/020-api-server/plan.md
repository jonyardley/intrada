# Implementation Plan: API Server

**Branch**: `020-api-server` | **Date**: 2026-02-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/020-api-server/spec.md`

## Summary

Build a REST API server for the Intrada music practice app, deployed on Fly.io with a Turso/libsql database backend. The server provides CRUD endpoints for pieces, exercises, and practice sessions, reusing validation rules from `intrada-core`. The database uses SQLite-compatible schema with JSON columns for tags (replacing Postgres TEXT arrays from the previous Shuttle-based API). CORS support enables the Cloudflare Workers frontend to communicate with the API. Health check endpoint supports Fly.io's built-in monitoring.

## Technical Context

**Language/Version**: Rust stable (1.89.0 in CI; workspace MSRV 1.75, axum 0.8 requires 1.78+)
**Primary Dependencies**: axum 0.8, libsql 0.9 (remote feature), libsql_migration 0.2.2 (content feature), tower-http 0.6 (cors), tokio 1, serde/serde_json 1, ulid 1, chrono 0.4, thiserror 1, tracing 0.1
**Storage**: Turso (managed libsql/SQLite) via HTTP protocol — `libsql` crate with `remote` feature
**Testing**: `cargo test` (unit tests in intrada-api crate + existing intrada-core tests)
**Target Platform**: Linux server (Fly.io Docker container, Debian bookworm-slim runtime)
**Project Type**: Workspace crate (`crates/intrada-api/`) — adds a new crate to the existing workspace
**Performance Goals**: Single-user app — no high-throughput requirements. Responses should feel instant (<500ms).
**Constraints**: Must not modify `intrada-core` or `intrada-web` crates. Must reuse validation from core.
**Scale/Scope**: Single user, hundreds of items. No pagination or search needed initially.

**Structure Decision**: The API server is a new workspace crate (`crates/intrada-api/`) following the existing pattern of `crates/intrada-core/` and `crates/intrada-web/`. This keeps the workspace organised and allows the API to depend on `intrada-core` for shared types and validation. The `Dockerfile` and `fly.toml` live at the workspace root since they build from the workspace context.
